# AI Hairstyle Try-On App — Engineering Design Doc

**Author:** Antigravity (Eng)
**Status:** Draft v0.1
**Last updated:** May 2026
**Reviewers:** TBD

---

## 1. Summary

We are building a single-page web application that allows users to upload a selfie or capture one via webcam, select a hairstyle preset, and instantly see an AI-generated transformation alongside a sassy stylist note. The system uses a FastAPI Python backend serving static HTML/JS/CSS, and integrates with the Google Gen AI SDK to concurrently call `gemini-3.1-flash-image-preview` (for image editing) and `gemini-3.5-flash` (for the structured stylist note). The most interesting engineering choice is executing these two model calls in parallel to aggressively minimize the user-facing latency.

## 2. Assumptions

- **Target scale:** <10k DAU in v1. No high-availability requirements beyond a single VM.
- **Latency budget:** p95 <5s for the core generation flow (from button tap to rendered result).
- **Platform:** Web-only, responsive (mobile and desktop browsers).
- **Cost ceiling:** <$0.05 per try-on.
- **Out of scope:** Accounts, session history, saving/sharing features, database storage.

## 3. Goals & non-goals

**Goals (v1):**
- Serve the single-page frontend rapidly.
- Handle base64 image uploads from the client.
- Call Gemini models concurrently (image generation + text evaluation).
- P95 latency for the `/api/tryon` endpoint <4.5s.

**Non-goals (v1):**
- Server-side database (no SQLite, no Postgres). All state is ephemeral.
- Image storage (no S3/GCS buckets). Images exist only in memory during the request and are returned as base64 to the client.
- Authentication and rate-limiting (assuming an internal or low-traffic beta deployment for now).
- Multi-region deployment.

## 4. Architecture

```mermaid
flowchart LR
  Client[Web Client (Vanilla JS)] -->|POST /api/tryon (base64 + preset)| API[FastAPI Server]
  API -->|prompt + image| GeminiImage[gemini-3.1-flash-image-preview]
  API -->|prompt + image| GeminiText[gemini-3.5-flash]
  GeminiImage -->|base64 edited image| API
  GeminiText -->|JSON (note + rating)| API
  API -->|merged JSON response| Client
```

**What's here:**
- **FastAPI Server** — Handles static file serving and the core `/api/tryon` endpoint.
- **Google Gen AI SDK** — Communicates with the models.

**What's deliberately NOT here:**
- **No Database** — We don't need to save the history. It's an ephemeral, one-shot experience.
- **No Redis/Celery Queue** — The latency target (<5s) means we can hold the HTTP request open safely. Async task queues add unnecessary complexity for a v1.
- **No React/Vue** — Vanilla HTML/CSS/JS is sufficient for a 2-state application and keeps the bundle size negligible.

## 5. Key components

### Web Client (Frontend)
- **Responsibility:** Capture/upload image, manage UI state (Input -> Loading -> Result), and render the response.
- **Tech choice:** Vanilla HTML/JS/CSS.
- **Why this choice:** Following the `moodjar-example` pattern. It’s lightweight, requires no build step, and forces us to keep the UI brutally simple.
- **Interface:** Calls `POST /api/tryon`.

### FastAPI Backend
- **Responsibility:** Validate inputs, coordinate parallel Gemini API calls, and format the response.
- **Tech choice:** Python + FastAPI + Uvicorn + `google-genai` SDK.
- **Why this choice:** FastAPI provides built-in async support (critical for parallel API calls) and Pydantic validation. The official `google-genai` SDK is required for Gemini 3.1 models.
- **Interface:** Exposes `POST /api/tryon` and `GET /api/health`.

## 6. Data model

Because we have no database, our data model consists entirely of Pydantic schemas for the API request/response.

```python
from pydantic import BaseModel
from typing import Literal

class TryOnRequest(BaseModel):
    # base64 encoded image string (data URI or raw base64)
    image: str 
    # One of the 4 supported presets
    preset_id: Literal["classic_bangs", "voluminous_afro", "neon_mohawk", "retro_mullet"]

class TryOnResponse(BaseModel):
    # base64 encoded transformed image
    transformed_image: str 
    # Sassy stylist note generated by Gemini
    stylist_note: str
    # Numeric or string vibe rating (e.g., "8/10")
    vibe_rating: str
```

## 7. API surface

### `POST /api/tryon`

- **Input:** `TryOnRequest` JSON.
- **Output:** `TryOnResponse` JSON.
- **Errors:** 
  - `400 Bad Request`: Invalid preset ID or malformed base64 image.
  - `500 Internal Server Error`: Gemini API timeout or failure.
- **Latency budget:** p95 <4.5s end-to-end. We achieve this by using `asyncio.gather` to make the image and text model calls simultaneously.

### `GET /api/health`

- **Input:** None
- **Output:** `{"status": "fabulous"}`
- **Latency budget:** <50ms.

## 8. Key trade-offs (with rejected alternatives)

### Decision: Parallel Model Calls vs. Single Multimodal Call

- **Chose:** Calling `gemini-3.1-flash-image-preview` for the image and `gemini-3.5-flash` for the text simultaneously.
- **Considered:** Using `gemini-3.1-flash-image-preview` with `response_modalities=["TEXT", "IMAGE"]` to do it in one call.
- **Why we picked this:** While a single call is theoretically cleaner, using `gemini-3.5-flash` for text allows us to enforce Strict Structured JSON Output (`response_mime_type="application/json"` with a Pydantic schema) to guarantee the note and rating fields exist. The image preview model might struggle to strictly adhere to complex JSON schemas while simultaneously editing an image. Parallelizing them via `asyncio.gather` incurs almost no latency penalty since they run concurrently.

### Decision: HTTP Request vs. WebSockets / Async Polling

- **Chose:** A standard synchronous-feeling HTTP request that holds open until the result is ready.
- **Considered:** Returning a `job_id` and having the client poll, or using WebSockets.
- **Why we picked this:** Gemini image generation is fast enough (typically 2-4s) that holding an HTTP connection open is safe. Polling adds significant frontend complexity and server-side state (we'd need a place to store the job status). We accept the risk of occasional client timeouts over building a complex async job system for v1.

## 9. Risks & unknowns

- **Image Generation Latency** — High. If the image model takes >10s, browsers or load balancers might timeout. *Mitigation: Monitor API latencies closely; if they drift, we must implement a polling architecture in v2.*
- **Model Refusals / Safety Filters** — Medium. Selfies might trigger safety filters if interpreted incorrectly. *Mitigation: Catch SDK `genai.errors.APIError` gracefully and return a polite 500 to the client ("The stylist is on break. Try another photo.")*
- **Base64 Payload Size** — Medium. Large smartphone photos can easily be 5MB+. Sending this as JSON base64 is heavy. *Mitigation: In the frontend JS, downscale images using a canvas element to max 1024x1024 before base64 encoding and sending.*

## 10. Testing strategy

This strategy focuses on validating the API contract and internal logic, mocking out the actual LLM calls.

**Unit tests (must have):**
- `test_image_downscaling`: Ensure the frontend (or a backend utility) correctly resizes and encodes/decodes images without corruption.
- `test_prompt_builder`: Verify that passing "neon_mohawk" correctly retrieves the exact configured prompt string for the image model.
- `test_json_schema_validation`: Ensure the `TryOnRequest` strictly rejects invalid `preset_id` values.

**Integration tests (one per happy path):**
- `test_tryon_api_success`: Send a dummy base64 string and a valid preset to `POST /api/tryon`. **MOCK** the `google-genai` SDK `client.models.generate_content` methods. Verify the endpoint returns `200 OK`, extracts the mocked image base64, and parses the mocked JSON text into a `TryOnResponse`.

**Deliberately not tested (and why):**
- **Model Output Quality:** We cannot unit test whether the generated mohawk looks "good" or if the stylist note is "sassy enough". This requires human verification.
- **Browser WebRTC API:** We will not write automated UI tests for the webcam capture flow, as headless browser camera permissions are notoriously flaky.

## 11. Rollout & monitoring

- **Rollout:** Deploy as a single container to Google Cloud Run or a similar serverless platform.
- **Monitoring:** Track `POST /api/tryon` 500 error rates (indicates model safety blocks or SDK timeouts) and p95 latency.
- **Rollback:** Standard container revision rollback.

## 12. Cost & capacity

- **Per-user cost:** 
  - `gemini-3.1-flash-image-preview`: ~$0.03 per image.
  - `gemini-3.5-flash`: Negligible (fractions of a cent for <100 output tokens).
  - Total: ~$0.03 per try-on.
- **Monthly budget at v1 scale (5,000 try-ons):** ~$150/month in API costs.
- **What breaks at 10× scale:** Fast parallel requests might hit the default Gemini API rate limits (RPM/TPM). We'd need to request a quota increase from Google Cloud.

## 14. Out of scope (will not do)

- **No persistent storage:** No database, no cloud storage buckets.
- **No auth service:** No logins, tokens, or session management.
- **No background job queue:** No Redis, Celery, or Cloud Tasks.
